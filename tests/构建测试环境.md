# WSL (Windows Subsystem for Linux)

Windows 10+ 系统支持 WSL (Windows Subsystem for Linux) 命令。

版本要求：Windows 10 版本 2004 (Build 19041) 及更高版本

**查看 wsl 版本**

```powershell
$ wsl --version

结果：
WSL 版本: 2.6.3.0
内核版本: 6.6.87.2-1
WSLg 版本: 1.0.71
MSRDC 版本: 1.2.6353
Direct3D 版本: 1.611.1-81528511
DXCore 版本: 10.0.26100.1-240331-1435.ge-release
Windows: 10.0.26200.7840
```
> [!Tip]
> 如果系统尚未安装 linux子系统，请按照提示进行安装。

**安装 wsl**

```powershell
$ wsl --install
```

> [!Warning]
> 需要使用管理员权限执行此命令。
> 此命令会自动启用必要的功能（wsl、虚拟机平台）、下载 WSL 2 内核并安装默认的 Ubuntu 发行版。安装后需要重启电脑。


# tox 3.x

**核心定义：** Tox 是一个自动化测试环境管理工具。

**实际使用场景：**

- 多版本兼容性测试（最核心场景）：

  - 痛点：你的代码在 Python 3.8 上跑通了，但用户可能用 3.9、3.10 或 3.11。你不想手动创建 4 个虚拟环境分别安装依赖跑测试。

  - 解决方案：配置一次，运行 tox，它自动创建 4 个隔离环境，分别安装依赖并运行测试，最后给你一份汇总报告。

- 确保测试环境纯净（隔离性）：

  - 痛点：你在本地开发时全局安装了一堆库，导致测试通过，但到了干净的 CI 服务器或用户电脑上就报错（依赖缺失或版本冲突）。

  - 解决方案：每次运行测试都从头创建一个全新的虚拟环境，安装指定版本的依赖，更有利于测试结果可复现。

- 统一团队开发流程：

  - 痛点：团队成员 A 用 pytest，成员 B 用 unittest，成员 C 忘记安装 flake8。

  - 解决方案：通过 tox.ini 锁定所有测试命令和依赖。新人只需运行 tox，即可执行全套标准检查（测试 + 代码风格 + 类型检查）。

- CI/CD 集成简化：

  - 痛点：在 GitHub Actions 或 Jenkins 中写复杂的脚本来管理环境。

  - 解决方案：CI 脚本只需一行 pip install tox && tox，复杂的环境逻辑全在 tox.ini 中。


**测试全部版本**
```powershell
$ tox
```

**测试指定版本**
```powershell
$ tox -e py311
```
